## 概述

先写测试，看它失败，再写最小代码让它通过。

**核心原则：** 如果你没有看到测试失败，你就不知道它是否在测试正确的东西。

> 本 skill 改编自 [obra/superpowers](https://github.com/obra/superpowers) 的 `test-driven-development` skill，根据 KQ-Forge "因地制宜"哲学调整为可选约束（非强制铁律）。

## 何时生效

当此 skill 被绑定到 Agent 的 `required_skills` 时，以下规则强制生效：

- 新功能实现
- 缺陷修复
- 行为变更

**例外（需人类确认）：**
- 一次性原型
- 生成代码（如 ORM migration）
- 纯配置文件修改

## 铁律

```
没有失败的测试，不写生产代码。
```

先写了代码再补测试？删掉代码，从测试开始。

- 不要保留代码作为"参考"
- 不要"一边看着代码一边写测试"
- 删除意味着删除

## RED-GREEN-REFACTOR 循环

### RED — 写一个失败的测试

写一个最小的测试，描述期望行为：

- 一个测试只测一个行为
- 测试名称清晰描述行为
- 尽量用真实代码，避免不必要的 mock

**写完后必须运行测试，确认：**
- 测试失败（不是报错）
- 失败原因是"功能缺失"（不是拼写错误）
- 失败信息符合预期

### GREEN — 写最小代码通过测试

写最简单的代码让测试通过：

- 不添加测试未要求的功能
- 不提前优化
- 不顺手重构其他代码

**写完后必须运行测试，确认：**
- 当前测试通过
- 其他测试仍然通过
- 无错误/警告

### REFACTOR — 清理

在绿灯状态下：
- 消除重复
- 改善命名
- 提取辅助函数

保持测试绿灯。不添加新行为。

### 重复

下一个失败测试，下一个功能。

## 验证清单

标记工作完成前：

- [ ] 每个新函数/方法都有测试
- [ ] 看到了每个测试失败
- [ ] 每个测试因正确原因失败（功能缺失，非拼写错误）
- [ ] 写了最小代码通过每个测试
- [ ] 所有测试通过
- [ ] 测试使用真实代码（mock 仅在不可避免时使用）
- [ ] 边界条件和错误场景已覆盖

## 反合理化

| 借口 | 现实 |
|------|------|
| "太简单不需要测试" | 简单代码也会出错。测试只需 30 秒 |
| "我先写完再补测试" | 事后通过的测试什么都证明不了——你没看到它失败 |
| "删掉 X 小时的工作太浪费" | 沉没成本谬误。保留无法信任的代码才是浪费 |
| "先探索一下再说" | 可以。但探索完后删掉代码，从 TDD 开始 |
| "测试太难写 = 设计没问题" | 恰恰相反。难测试 = 难使用 = 设计有问题 |
| "TDD 太教条了" | TDD 是务实的：比事后调试更快发现 bug |
| "手动测试过了" | 手动测试是临时的，无法重复，无法证明边界条件 |

## 红旗 — 停下来

出现以下情况时，停下来重新开始：

- 先写了代码再写测试
- 测试立即通过（说明在测试已有行为）
- 无法解释测试为什么失败
- 想着"就这一次跳过 TDD"
- 想保留代码"作为参考"

## 与 debug skill 的集成

发现 bug 时：先写一个复现 bug 的失败测试，然后按 TDD 循环修复。测试既证明了修复有效，又防止了回归。

## 注意事项

此 skill 是**可选约束**。项目可以选择：
- 绑定到 implementer 的 `required_skills` → 强制 TDD
- 放在 `optional_skills` → 按需使用
- 不绑定 → 不生效

不同项目、不同阶段可以有不同选择。这是"因地制宜"。
