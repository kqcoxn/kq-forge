## 何时使用

- 任务完成后（无论成功还是失败）
- 错误发生后（强制触发，如果 config 中 reflect_on_error: true）
- round_cap 耗尽后（对抗未收敛时）
- 用户手动触发 `/remember` 时

## 核心原则

1. **只沉淀非显而易见的知识**——"变量要命名清楚"不值得记录，"这个项目的 auth 模块改动必须跑集成测试"值得记录
2. **区分根因类型**——不同类型的根因需要不同的应对策略
3. **具体胜于抽象**——"下次注意"没有价值，"修改 payment 模块时必须同步更新 webhook handler"有价值
4. **记忆是规则，范式是能力**——记忆告诉你"不能做什么/必须做什么"，范式告诉你"应该怎么做"

## 反思流程

### 第一步：事实记录

客观描述发生了什么：
- 任务是什么？
- 结果是什么？（成功/失败/部分完成）
- 过程中出现了什么意外？

**只记录事实，不做评判。**

### 第二步：根因分析

如果出现了问题或意外，分析根因属于哪个类型：

| 根因类型 | 含义 | 应对策略 |
|---------|------|---------|
| **知识不足** | 不知道某个事实或规则 | 沉淀为 memory/facts.md 或 memory/rules.md |
| **流程缺失** | 没有对应的检查步骤或约束 | 沉淀为 memory/rules.md 或新增 skill |
| **沟通偏差** | 理解与意图不一致 | 沉淀为 memory/lessons.md，改进 AC 定义方式 |
| **工具限制** | 工具能力不足导致 | 沉淀为 memory/facts.md，标记为已知限制 |
| **复杂度超预期** | 任务比预估更复杂 | 沉淀为 memory/lessons.md，改进分解策略 |

**如果任务顺利完成且无意外**——记录"本次执行顺利"作为基线，不强制找问题。

### 第三步：沉淀决策

根据根因类型决定沉淀到哪里：

#### 沉淀为记忆（Memory）

写入 `.kqforge/memory/` 对应文件：

- **rules.md**：项目规则与约束
  - 格式：`- [日期] 规则内容`
  - 示例：`- [2025-01-15] 修改 auth 模块后必须运行完整集成测试`

- **facts.md**：项目事实
  - 格式：`- [日期] 事实内容`
  - 示例：`- [2025-01-15] payment 服务使用 Stripe API v3，webhook 签名验证在 middleware 层`

- **lessons.md**：教训
  - 格式：`- [日期] 教训内容（附上下文）`
  - 示例：`- [2025-01-15] 不要在 transaction 内调用外部 API——上次导致死锁（见 PR #42）`

#### 提炼为范式（Paradigm）

当你发现以下信号时，应建议用户创建范式：

- 同一模式出现 **3 次以上**
- 某个领域形成了稳定的操作流程
- 某类问题有了成熟的解决套路

**创建范式前必须征得用户同意。**

范式文件直接创建在平台 agents 目录中（如 `.opencode/agents/paradigm-{name}.md`），文件名以 `paradigm-` 为前缀。格式与普通 Agent 相同，body 包含四个段落：

```markdown
---
name: paradigm-{name}
description: [模式的一句话描述]
---

## 模式

[抽象的工作模式描述]

## 步骤

1. [标准操作流程步骤 1]
2. [标准操作流程步骤 2]
...

## 判断依据

- [何时应用该模式的条件]

## 反例

- [不应使用该模式的场景]
```

### 第四步：验证沉淀质量

沉淀前自检：

- [ ] 内容是否**具体可操作**？（不是"要注意"而是"必须做X"）
- [ ] 内容是否**非显而易见**？（新人看到会觉得有价值）
- [ ] 内容是否**不重复**？（memory 中没有类似条目）
- [ ] 分类是否正确？（规则 vs 事实 vs 教训）

## 沉淀标准

### 值得沉淀的

- 项目特有的规则和约定（非通用知识）
- 踩过的坑和对应的防范措施
- 模块间的隐含依赖关系
- 工具/框架的已知限制和绕过方法
- 成功模式的关键步骤

### 不值得沉淀的

- 通用编程知识（"函数应该职责单一"）
- 一次性的临时信息（"今天服务器重启了"）
- 过于细节的实现记录（"在第 42 行加了一个 if"）
- 已经在代码注释中说明的内容

## 范式提炼信号

当你在反思中发现以下模式时，在报告中标注"建议创建范式"：

```markdown
### 范式提炼建议
- 模式名称：[建议名称]
- 出现次数：[N 次]
- 模式描述：[这个项目的 X 总是按照 Y 流程来做]
- 相关记忆：[引用 memory 中的相关条目]
- 目标文件：`{平台agents目录}/paradigm-{name}.md`
```

**注意**：提出建议后必须等待用户确认，不可自行创建范式文件。

## 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 反思流于形式 | "下次注意"没有任何指导价值 | 必须具体到可操作的规则或事实 |
| 沉淀太泛 | "要仔细"对任何人都没帮助 | 具体到场景、条件、操作 |
| 不区分根因类型 | 所有问题都归为"不够仔细" | 按类型分析，对症下药 |
| 重复沉淀 | memory 文件膨胀，信噪比下降 | 沉淀前检查是否已有类似条目 |
| 强制找问题 | 顺利的任务也硬凑教训 | 顺利就记录"顺利"，不强制 |

## 输出格式

```markdown
## 反思记录

### 事实
- 任务：[描述]
- 结果：[成功 / 失败 / 部分完成]
- 意外：[有 / 无]

### 根因分析（如有意外）
- 根因类型：[知识不足 / 流程缺失 / 沟通偏差 / 工具限制 / 复杂度超预期]
- 具体原因：[描述]

### 沉淀
- 写入：memory/[category].md
- 内容：`- [日期] [具体内容]`

### 范式提炼建议（如适用）
- 建议：/evolve [name]
- 理由：[为什么认为这是一个可提炼的模式]
```
