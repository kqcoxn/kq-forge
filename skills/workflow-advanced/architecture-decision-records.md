# 架构决策记录（ADR）

## 核心理念

> 决策的理由比决策本身更重要。记录"为什么"，让未来的开发者理解代码库为何是这个形状。

## 触发时机

### 显式触发

- "记录这个决策" / "ADR this"
- "我们决定用 X 而不是 Y，因为..."
- "为什么我们选了 X？"（查询已有 ADR）

### 隐式触发（建议记录，不自动创建）

- 比较两个框架/库并得出结论
- 做出数据库 schema 设计选择
- 在架构模式间做出选择（单体 vs 微服务、REST vs GraphQL）
- 确定认证/授权策略
- 选择部署基础设施

## ADR 格式

```markdown
# ADR-NNNN: [决策标题]

**日期**: YYYY-MM-DD
**状态**: 提议中 | 已接受 | 已废弃 | 被 ADR-NNNN 取代
**决策者**: [参与人]

## 背景

是什么问题/情况促使了这个决策？

[2-5 句话描述情况、约束和作用力]

## 决策

我们决定做什么？

[1-3 句话清晰陈述决策]

## 考虑过的替代方案

### 替代方案 1：[名称]
- **优势**: [好处]
- **劣势**: [缺点]
- **未选原因**: [具体原因]

### 替代方案 2：[名称]
- **优势**: [好处]
- **劣势**: [缺点]
- **未选原因**: [具体原因]

## 后果

### 正面
- [好处 1]
- [好处 2]

### 负面
- [权衡 1]
- [权衡 2]

### 风险
- [风险及缓解措施]
```

## 工作流程

### 创建新 ADR

1. **初始化（首次）**：如果 `docs/adr/` 不存在，征求用户同意后创建目录、README 索引和模板
2. **识别决策**：提取核心架构选择
3. **收集背景**：什么问题促使了这个决策？有什么约束？
4. **记录替代方案**：考虑过哪些选项？为什么被否决？
5. **陈述后果**：权衡是什么？什么变容易/变难了？
6. **分配编号**：扫描已有 ADR，递增编号
7. **确认并写入**：呈现草稿给用户审阅，获得明确批准后才写入文件
8. **更新索引**：追加到 `docs/adr/README.md`

### 查询已有 ADR

1. 检查 `docs/adr/` 是否存在
2. 扫描索引查找相关条目
3. 读取匹配的 ADR 文件，呈现背景和决策部分
4. 未找到则询问是否要现在记录

## 目录结构

```
docs/
└── adr/
    ├── README.md              ← 所有 ADR 的索引
    ├── 0001-use-nextjs.md
    ├── 0002-postgres-over-mongo.md
    └── template.md            ← 空白模板供手动使用
```

## ADR 生命周期

```
提议中 → 已接受 → [已废弃 | 被 ADR-NNNN 取代]
```

## 值得记录的决策类别

| 类别 | 示例 |
|------|------|
| 技术选型 | 框架、语言、数据库、云服务商 |
| 架构模式 | 单体 vs 微服务、事件驱动、CQRS |
| API 设计 | REST vs GraphQL、版本策略、认证机制 |
| 数据建模 | Schema 设计、范式化决策、缓存策略 |
| 基础设施 | 部署模型、CI/CD 流水线、监控栈 |
| 安全 | 认证策略、加密方案、密钥管理 |
| 测试 | 测试框架、覆盖率目标、E2E vs 集成比例 |
| 流程 | 分支策略、审查流程、发布节奏 |

## 好 ADR 的标准

### 应该

- **具体** — "使用 Prisma ORM" 而非 "使用某个 ORM"
- **记录为什么** — 理由比结论重要
- **包含被否决的方案** — 未来开发者需要知道考虑过什么
- **诚实陈述后果** — 每个决策都有权衡
- **保持简短** — 一个 ADR 应该 2 分钟内读完
- **使用现在时** — "我们使用 X" 而非 "我们将使用 X"

### 不应该

- 记录琐碎决策（变量命名、格式化偏好）
- 写成论文（背景超过 10 行就太长了）
- 省略替代方案（"就是选了它"不是有效理由）
- 不标注回填（如果记录过去的决策，注明原始日期）
- 让 ADR 过时（被取代的决策应引用替代者）
