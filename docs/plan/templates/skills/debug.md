## 何时使用

- 代码行为与预期不符时
- 测试失败且原因不明时
- 生产环境出现异常时
- 实现过程中遇到意外行为时

## 核心原则

1. **先复现，再定位**——不能稳定复现的 bug 不要猜测修复
2. **一次只改一个变量**——同时改多处无法确定哪个修复生效
3. **假设驱动**——每次排查都基于明确的假设，验证后再进入下一步
4. **3 次失败就退一步**——连续 3 次假设验证失败，质疑问题定义本身

## 调试流程

### 第一步：复现

**目标：** 建立稳定的复现路径

1. 明确"预期行为"和"实际行为"的差异
2. 找到最小复现步骤（去除无关变量）
3. 确认复现是稳定的（多次执行结果一致）

**如果无法复现：**
- 检查是否是环境差异（开发 vs 生产、OS 差异）
- 检查是否是时序相关（并发、定时器）
- 检查是否是数据相关（特定输入才触发）
- 如果仍无法复现 → 记录现象，添加日志/监控，等待下次出现

### 第二步：定位

**目标：** 缩小问题范围到具体代码位置

**方法 A：二分法**
- 在代码执行路径的中间点添加检查
- 确认问题在前半段还是后半段
- 递归缩小范围直到定位到具体行

**方法 B：日志追踪**
- 在关键节点添加日志输出
- 追踪数据流转过程
- 找到数据从"正确"变为"错误"的转折点

**方法 C：最小用例**
- 构造最简单的输入触发问题
- 逐步添加复杂度，找到触发条件的边界

### 第三步：假设

**目标：** 形成可验证的根因假设

基于定位结果，提出假设：
- "问题是因为 X 条件下 Y 变量没有被正确初始化"
- "问题是因为 A 函数在并发调用时没有加锁"

**假设必须是可验证的**——能通过具体操作确认真假。

### 第四步：验证

**目标：** 确认或否定假设

- 设计一个实验来验证假设
- 实验应该能明确区分"假设成立"和"假设不成立"
- 执行实验，记录结果

**如果假设成立** → 进入修复步骤
**如果假设不成立** → 回到第三步，提出新假设

### 第五步：修复

**目标：** 消除根因

- 修复应该针对根因，而非症状
- 修复范围最小化
- 修复后重新执行复现步骤，确认问题消失

### 第六步：回归

**目标：** 确认修复没有引入新问题

- 运行相关测试
- 检查修复是否影响了其他功能
- 如果有回归 → 修复方案需要调整

## 升级机制

### 3 次假设失败 → 退一步

如果连续 3 次假设验证都失败：

1. **停下来**，不要继续猜
2. **质疑问题定义**：
   - "预期行为"真的是正确的吗？
   - 问题是否被错误描述了？
   - 是否在错误的层面寻找原因？
3. **扩大视野**：
   - 检查更上层的调用链
   - 检查环境/配置因素
   - 检查是否是多个问题叠加
4. 如果仍无法定位 → 返回 BLOCKED，请求人类协助

### 修复失败 → 质疑架构

如果修复后问题仍然存在或引入了新问题：

1. 考虑是否是架构层面的问题（而非代码层面）
2. 考虑是否需要更大范围的重构
3. 返回 DONE_WITH_CONCERNS，说明情况

## 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 不复现就猜 | 修了不存在的 bug，或修了症状没修根因 | 先建立稳定复现 |
| 同时改多处 | 不知道哪个修复生效，可能引入新问题 | 一次只改一个变量 |
| 修了症状没修根因 | 问题反复出现 | 追问"为什么会出现这个症状" |
| 无限循环猜测 | 浪费时间，越改越乱 | 3 次失败就退一步 |
| 修复后不验证 | 以为修好了实际没有 | 必须重新执行复现步骤 |

### 修复方案
- 修改内容：...
- 修改理由：...

### 回归验证
- 原始问题：已消失 ✓
- 相关功能：未受影响 ✓
