## 何时使用

- 收到新任务需要分解为子任务时
- 需要做技术方案选型时
- 面对模糊需求需要澄清和结构化时
- 需要评估多个实现路径的优劣时

## 核心方法

### 第一步：目标澄清

在动手分解之前，先确认以下问题：

1. **最终期望结果是什么？**（用一句话描述）
2. **谁是受益者？**（用户？开发者？系统？）
3. **有哪些硬约束？**（时间、技术栈、兼容性等）
4. **成功的标准是什么？**（可验证的条件）

如果以上任何一个问题无法回答 → 返回 NEEDS_CONTEXT，不要在模糊中前进。

### 第二步：验收标准定义

将成功标准转化为可判定的 Acceptance Criteria（AC）：

- 每条 AC 必须是**二元的**（满足/不满足，没有"部分满足"）
- 每条 AC 必须是**可验证的**（能通过具体操作确认）
- AC 数量建议 3-7 条（太少覆盖不全，太多难以管理）

**好的 AC 示例：**
- ✅ "用户点击提交按钮后，表单数据保存到数据库且页面显示成功提示"
- ✅ "API 响应时间 < 200ms（P95）"

**差的 AC 示例：**
- ❌ "代码质量好"（不可判定）
- ❌ "性能可以接受"（没有具体标准）

### 第三步：方案探索

如果存在多种实现路径：

1. 列出所有可行方案（至少 2 个）
2. 对每个方案评估：
   - 复杂度（实现难度）
   - 风险（可能出什么问题）
   - 可维护性（未来修改的成本）
   - 与现有架构的契合度
3. 选择最优方案并说明理由
4. 如果无法确定 → 标记为需要人类决策的节点

### 第四步：任务分解

将选定方案拆解为子任务：

**分解原则：**
- 每个子任务应在**单个 Agent 的单次执行**中可完成
- 每个子任务有明确的**输入**和**预期输出**
- 子任务之间的**依赖关系**必须明确
- 可并行的子任务应标记为可并行

**分解粒度（由 autonomy 等级决定）：**
- L0：最细粒度，每个决策点都是独立子任务
- L1：中等粒度，关键决策暴露，机械步骤合并
- L2：粗粒度，只在 milestone 级别拆分
- L3：最粗粒度，整块工作作为单个任务

**子任务模板：**
```
子任务 N：[标题]
- 委派给：[agent 名称]
- 输入：[需要什么信息/文件]
- 预期输出：[产出什么]
- AC：[该子任务的验收标准]
- 依赖：[依赖哪些前置子任务]
```

## 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 分解太粗 | 子任务内部还有隐含决策，执行者被迫自行判断 | 确保每个子任务无歧义 |
| 分解太细 | 机械步骤被拆成独立任务，增加编排开销 | 合并纯机械步骤 |
| 遗漏依赖 | 并行执行时出现冲突或缺少前置条件 | 画依赖图，确认无环且完整 |
| AC 不可验证 | 无法客观判定任务是否完成 | 每条 AC 用"如何验证"来检验 |
| 跳过方案探索 | 选了次优方案，后期返工 | 至少考虑 2 个方案再决定 |

## 输出格式

```markdown
## 设计方案

### 目标
[一句话]

### 约束
- [约束1]
- [约束2]

### 验收标准
- [ ] AC1: ...
- [ ] AC2: ...
- [ ] AC3: ...

### 方案选择
[选定方案及理由，如有多个候选方案简述对比]

### 子任务分解
1. [子任务1] → implementer
   - 输入：...
   - 输出：...
   - AC：...
2. [子任务2] → implementer
   - 输入：...
   - 输出：...
   - AC：...
   - 依赖：子任务1

### 依赖关系
子任务1 → 子任务2 → 子任务3
子任务1 → 子任务4（可与子任务2并行）
```
