## 何时使用

- 需要编写新代码时
- 需要修改现有代码时
- 需要修复缺陷时
- 需要执行技术方案时

## 核心流程

### 第一步：理解需求

在写任何代码之前：

1. **阅读任务描述和 AC**——确保理解"做什么"和"怎样算完成"
2. **阅读相关代码**——理解现有架构、约定、依赖关系
3. **确认可行性**——如果发现 AC 不可行或信息不足，立即返回 NEEDS_CONTEXT

**关键原则：先读再写，永远不凭记忆假设文件内容。**

### 第二步：实现

遵循以下原则编写代码：

**最小变更原则：**
- 只改需要改的，不顺手重构
- 不引入不必要的新依赖
- 不改变不相关的代码格式

**风格一致原则：**
- 遵循项目现有的命名约定
- 遵循项目现有的文件组织方式
- 遵循项目现有的错误处理模式
- 如果项目没有明确约定，遵循语言社区的主流约定

**防御性编码原则：**
- 处理边界条件（空值、空数组、异常输入）
- 不硬编码敏感信息
- 资源获取后确保释放（文件句柄、数据库连接等）

### 第三步：自检

实现完成后，对照以下清单逐项确认：

- [ ] 代码可编译/可运行，无语法错误
- [ ] 所有 AC 逐条满足
- [ ] 变更范围最小化，没有无关改动
- [ ] 遵循项目现有代码风格
- [ ] 没有引入新的 lint 警告/错误
- [ ] 没有硬编码的敏感信息
- [ ] 边界条件已考虑
- [ ] 错误处理已覆盖关键路径

### 第四步：报告

返回状态时附带：
- 变更文件列表
- 变更摘要（做了什么、为什么这样做）
- 如有顾虑，明确说明

## 质量底线

以下是**不可协商的底线**，无论任务多紧急都必须满足：

1. **代码可运行**：不留语法错误、不留未解决的导入
2. **AC 满足**：每条 AC 都经过自检确认
3. **无敏感信息泄露**：密钥、密码、token 不出现在代码中
4. **scope 内操作**：不修改管辖范围外的文件

底线之上的质量标准（如测试覆盖、性能优化）由项目配置的 skills 决定。

## 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 不读现有代码就开始写 | 风格不一致、重复造轮子、破坏现有逻辑 | 先花时间阅读相关文件 |
| 改动范围超出任务要求 | 引入意外副作用、增加 review 负担 | 严格限制在 AC 要求的范围内 |
| 引入不必要的依赖 | 增加维护成本、潜在安全风险 | 优先用现有工具解决 |
| 忽略边界条件 | 生产环境出现意外崩溃 | 主动思考"如果输入是空/null/超大会怎样" |
| 凭记忆假设文件内容 | 基于过时信息编码，导致冲突或错误 | 每次都重新读取文件 |
| AC 不合理时自行决定 | 交付物与期望不符 | 返回 NEEDS_CONTEXT 让 lead 澄清 |

## 与其他 Skills 的关系

- **debug**：当实现过程中遇到意外行为时，切换到 debug 流程
- **refactor**：如果发现现有代码需要重构才能实现需求，先确认是否在任务范围内
- **test-first**（可选）：如果项目绑定了此 skill，则必须先写测试再写实现
