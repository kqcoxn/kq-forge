## 概述

Git 历史是项目的记忆。清晰的提交历史让回溯、审查和回滚都变得简单。

> 本 skill 综合参考了 [OmO/git-master](https://github.com/code-yeongyu/oh-my-openagent) 的原子提交理念和 [ECC](https://github.com/affaan-m/everything-claude-code) 的 git-workflow 规范。

## 提交信息格式

### 检测优先

**在写提交信息之前，先检测项目现有风格：**

```bash
git log -20 --oneline
```

如果项目已有明确风格（如 Conventional Commits），遵循项目风格。
如果没有明确风格，使用以下默认格式：

### 默认格式

```
<type>: <描述>

<可选正文>
```

**类型：**
- `feat`: 新功能
- `fix`: 缺陷修复
- `refactor`: 重构（不改变行为）
- `docs`: 文档
- `test`: 测试
- `chore`: 杂务（构建、依赖等）
- `perf`: 性能优化
- `ci`: CI/CD 配置

**描述规则：**
- 使用项目的主要语言（中文项目用中文，英文项目用英文）
- 简洁明了，说明"做了什么"
- 不超过 72 字符

## 原子提交原则

### 核心规则

**一个提交 = 一个逻辑变更。**

| 文件数 | 最少提交数 |
|--------|-----------|
| 1-3 文件 | 1 个提交 |
| 4-6 文件 | 至少 2 个提交 |
| 7-10 文件 | 至少 3 个提交 |
| 10+ 文件 | 至少 5 个提交 |

### 拆分依据

| 条件 | 动作 |
|------|------|
| 不同目录/模块 | 拆分 |
| 不同组件类型（model/service/view） | 拆分 |
| 可以独立回滚 | 拆分 |
| 不同关注点（UI/逻辑/配置/测试） | 拆分 |
| 新文件 vs 修改文件 | 拆分 |

### 合并条件

只有当**以下所有条件都满足**时才合并到一个提交：
- 是同一个原子单元（如函数 + 它的测试）
- 拆分会导致编译失败
- 能用一句话解释为什么必须在一起

### 实现与测试配对

**测试文件必须和实现文件在同一个提交中。**

```
✅ feat: 添加用户注册功能
   - src/api/register.ts
   - src/api/register.test.ts

❌ feat: 添加用户注册功能
   - src/api/register.ts
   （测试在另一个提交里）
```

## 分支规范

### 分支命名

```
<type>/<简短描述>
```

示例：
- `feat/user-registration`
- `fix/auth-token-expiry`
- `refactor/payment-module`

### 安全规则

| 规则 | 说明 |
|------|------|
| 永远不在 main/master 上 rebase | 只能新增提交 |
| force push 前必须确认 | 使用 `--force-with-lease` 而非 `--force` |
| 不 rewrite 已推送的历史 | 除非明确获得许可 |
| 提交前检查敏感信息 | 密钥、密码、token 不入库 |

## 提交前检查清单

每次提交前确认：

- [ ] 提交信息符合项目风格
- [ ] 一个提交只包含一个逻辑变更
- [ ] 测试文件与实现文件在同一提交
- [ ] 没有包含敏感信息（密钥、密码等）
- [ ] 没有包含临时文件（.env.local、debug 日志等）
- [ ] 变更可以独立理解（不依赖未提交的其他变更）

## PR 工作流

创建 PR 时：

1. 分析完整的提交历史（不只是最新提交）
2. 使用 `git diff [base-branch]...HEAD` 查看所有变更
3. 撰写全面的 PR 摘要
4. 包含测试计划
5. 新分支使用 `-u` 标志推送

## 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 一个大提交包含所有变更 | 无法独立回滚、难以审查 | 按逻辑拆分为多个原子提交 |
| 提交信息模糊（"更新代码"） | 历史无法追溯 | 明确说明做了什么 |
| 测试和实现分开提交 | 中间状态编译失败 | 配对在同一提交 |
| 提交了敏感信息 | 安全风险（即使后续删除，历史中仍存在） | 提交前检查 |
| 不检测项目风格就用默认格式 | 风格不一致 | 先 `git log` 检测 |
