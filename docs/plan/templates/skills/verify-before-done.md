## 概述

宣称完成而不验证，是不诚实，不是高效。

**核心原则：** 先有证据，再有结论。永远如此。

> 本 skill 改编自 [obra/superpowers](https://github.com/obra/superpowers) 的 `verification-before-completion` skill。

## 铁律

```
没有新鲜的验证证据，不能宣称完成。
```

如果你在这条消息中没有运行验证命令，你就不能声称它通过了。

## 门控流程

在宣称任何状态或表达满意之前：

1. **识别**：什么命令能证明这个声明？
2. **运行**：执行完整的验证命令（新鲜的、完整的）
3. **阅读**：完整输出，检查退出码，计数失败
4. **确认**：输出是否支持你的声明？
   - 如果**否**：陈述实际状态 + 证据
   - 如果**是**：陈述声明 + 证据
5. **然后**：才能做出声明

跳过任何一步 = 说谎，不是验证。

## 常见场景

| 声明 | 需要的证据 | 不充分的证据 |
|------|-----------|-------------|
| "测试通过" | 测试命令输出：0 failures | 上次运行结果、"应该能过" |
| "lint 干净" | lint 输出：0 errors | 部分检查、推测 |
| "构建成功" | 构建命令：exit 0 | lint 通过、日志看起来正常 |
| "bug 已修复" | 原始症状测试：通过 | 代码改了、假设修好了 |
| "回归测试有效" | RED-GREEN 循环验证 | 测试通过一次 |
| "需求满足" | 逐条对照 AC 检查 | 测试通过 |

## 红旗 — 停下来

出现以下情况时，你正在违反此规则：

- 使用"应该"、"大概"、"看起来"
- 在验证之前表达满意（"好了！"、"完美！"、"搞定！"）
- 准备提交/推送/PR 但没有验证
- 信任子 Agent 的成功报告而不独立验证
- 依赖部分验证
- 想着"就这一次"
- 疲惫了想赶紧结束
- **任何暗示成功但没有运行验证的措辞**

## 反合理化

| 借口 | 现实 |
|------|------|
| "应该没问题了" | 运行验证命令 |
| "我很有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "lint 过了" | lint ≠ 编译器 ≠ 测试 |
| "Agent 说成功了" | 独立验证 |
| "太累了" | 疲惫 ≠ 借口 |
| "部分检查够了" | 部分证明不了什么 |

## 验证模式

**测试：**
```
✅ [运行测试命令] → [看到: 34/34 pass] → "所有测试通过"
❌ "应该能过了" / "看起来对了"
```

**构建：**
```
✅ [运行构建] → [看到: exit 0] → "构建通过"
❌ "lint 过了"（lint 不检查编译）
```

**需求：**
```
✅ 重读 AC → 逐条对照 → 验证每条 → 报告结果
❌ "测试过了，任务完成"
```

**子 Agent 委派：**
```
✅ Agent 报告成功 → 检查实际变更 → 独立验证 → 报告实际状态
❌ 信任 Agent 报告
```

## 适用时机

**在以下情况之前必须验证：**
- 任何形式的完成/成功声明
- 任何满意的表达
- 提交、PR 创建、任务标记完成
- 进入下一个任务
- 返回 DONE 状态

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后才能声称结果。

这是不可协商的。
